Stageverslagen

under Software

Verslag 11-01

Om da dag te beginnen heeft Matthias ons eerst een korte rondleiding gegeven. Daarna heeft hij ons op zijn bureau wat uitleg gegeven over reguliere expressies omdat we dit zullen nodig hebben om onze testen te schrijven. Dan mochten we zelf aan de slag met de QF-Test tool. Onze opdracht was om lege velden te vullen met testgegevens. Alles ging vlot behalve het laatste uur waren we aan het sukkelen met de IF-syntax.

Verslag 12-01

Gisteren zaten we vast aan een if, we kregen dit niet voor vijf uur opgelost dus dit moesten we deze morgen als eerste aanpakken. Na een dik halfuur vonden dan eindelijk de oplossing en vanaf dan ging het vrij vlot tot we een error(modal dialog) kregen die we niet konden oplossen. Wim zei ons dat we dat stukje mochten overslaan en beginnen aan het maken van groepen. Dit ging ook wel vlot, hier hadden geen problemen mee. Morgen moeten we het stukje van de groepen nog afwerken en verfijnen. Dan kunnen we eventueel aan onze oefeningen van reguliere expressies beginnen.

Verslag 13-01

Vandaag hebben aan de test van CreateGroups verder gewerkt. Eerst moesten we nog kleine details aanpassen, dan hebben we al onze code overzichtelijk gesorteerd. Dan hebben we nog een probleem in de test van CreateFreeFields opgelost, dat was de modal dialog error van gisteren. Nu zijn ook onze eerste twee testen zo goed als afgewerkt. Morgen zullen we waarschijnlijk een nieuwe opdracht krijgen van Wim of we kunnen ook aan de reguliere expressies werken want daar hadden we vandaag geen tijd voor.

Verslag 14-01

Vandaag hebben we onze twee grote testen(CreateFreeFields en CreateGroups) afgewerkt. Dan zijn we begonnen aan de taak over reguliere expressies, de eerste oefeningen gingen allemaal heel vlot en we hadden alles ook van de eerste keer juist. Alleen met de laatste oefening hadden we veel last, we waren vergeten dat als je een groep maakt deze automatisch wordt weggeschreven. Nadat we het antwoord hadden gevonden met hulp van Matthias heeft Wim ons een opdracht gegeven om de lijst van bezoekers aan te vullen, maar dan bleek dat deze al voorgemaakt was. Nu zijn we bezig met het met een test te schrijven die bezoeken aanmaakt en daar gaan we ook morgen aan verder werken.

verslag 15-01

Vandaag hebben we verder gewerkt aan de test om bezoeken aan te vullen(CreateVisits). Deze test was gelijkaardig op te lossen aan de twee tests die we al hadden geschreven. Het enige verschil was dat je voor deze test een script nodig had om de input van een uur gelijk te stellen aan de huidige tijd. We hebben geprobeerd om dit zelf te schrijven maar dit is ons dan niet gelukt. Wim heeft ons het script gegeven, we hebben hier dan wat mee gespeeld om nog andere scriptjes te schrijven die hierop gebaseerd waren en die de huidige tijd plus vijf dagen returned. Maandag gaan we ook nog een script schrijven dat hetzelfde doet maar dan me

Verslag 18-01

Vandaag hebben we een nieuwe grote opdracht gekregen. We moesten met de tweede applicatie(PrimeVisit) aan het werk. Onze opdracht was om bezoeken te simuleren in opzicht van een klant of bezoeker, dit ging allemaal vlot en we hebben hier geen grote problemen ondervonden. Daarna moesten we terug met PrimeTime aan het werk. We moesten de VisitorFields vullen met voorgemaakte gegevens.

Verslag 19-01

Gisteren waren we nog maar net begonnen aan het vullen van de bezoekers velden dus we konden hier direct aan verder werken. Dit was wel een grote test. We kregen een aantal keer een error dat er een knop niet gevonden kon worden. We dachten eerst dat dit aan ons lag maar toch kregen we het niet opgelost en dus vroegen we hulp Wim kon hier ook geen oplossing voor vinden. Later bleek dan dat het een fout was in QF-Test we konden dus niets anders doen dan het gewoon negeren.

Verslag 20-01

Vandaag zijn we begonnen met het afwerken van onze test van vorige dagen, hier hebben we geen problemen meer mee ondervonden. In de namiddag kregen we de opdracht om leveringen en bezoeken te simuleren. Hierbij moesten we het veld van een datum vullen maar we moesten zorgen dat die datum altijd gelijk bleef aan de huidige datum. Dit moesten we dus met een groovy script doen maar we wisten niet echt hoe we hieraan moesten beginnen. We hebben veel output gekregen maar nooit echt degene die we zochten dus hier moeten we morgen zeker en vast eerst mee voort gaan.

Verslag 21-01

We hebben vandaag nog langen zitten zwoegen aan ons script maar uiteindeijk is het ons dan toch gelukt. In de namiddag moesten we een test schrijven die de gebruiker vanzelf uitlogged(UserLogout) hier hadden we geen problemen mee en dan hebben we om de dag af te sluiten al de voorheen gemaakte testen geordend en een toepasselijke naam gegeven.

Verslag 22-02

Vandaag was de laatste dag en we kregen vanmorgen te horen dat Matthias graag zou hebben dat we een kleine presentatie deden over de voorbije twee weken. Om te laten zien wat we allemaal gedaan hadden in die twee weken en hoe het voor ons verlopen was. We zijn begonnen met het afwerken van alle testen, ook hebben we hier en daar nog een detail aangepast. Daarna zijn we direct begonnen aan onze presentatie die we dan in de namiddag moesten houden voor een paar leden van het GET N.V-team.

Workflow: Hugo vs WordPress

under Software

Wordpress

Voordelen:

  • Het is een open source programma dus je kan je code baseren op anderen hun code.
  • Het is makkelijk om te gebruiken, je kan een hele site maken met alleen maar muisklikken zonder zelf code te hoeven schrijven.
  • Je kan van een basissite beginne en dan je eigen onderdelen toevoegen.

Nadelen:

  • Omdat het een open source programma is is het ook makkelijker voor hackers om beveiligingslekken te vinden.
  • Je kan het niet gebruiken om geavanceerde sites te maken.
  • Je hebt niet veel variatie tussen de verschillende sites.

Hugo

Voordelen:

  • Omdat de code is onderverdeeld in verschillende mappen is het heel overzichtelijk.
  • Het template-systeem zorgt ervoor dat je niet alle code zelf moet schrijven maar dat je toch volledige controle hebt over wat je doet.
  • Je kunt je baseren op bestaande themes en vanaf daar beginnen aanpassen.

Nadelen:

  • Moeilijk voor beginners om het mappensysteem te begrijpen.
  • Er is niet veel over te vinden op het internet.
  • Moeilijk om zelf code te sschrijven als je van een theme begint.

Directory-structuur:

De code van mijn Hugo site is onderverdeeld in verschillende mappen en bestanden om het overzichtelijk te houden. Hieronder een opsomming van mijn mappen:

  • Content
  • Layouts
  • Static
  • En nog een config.toml bestand

Content

In deze map zitten al mijn posts. Dat zijn Markdown bestanden die ik zelf geschreven heb. Deze bestanden zullen dan ingedeeld worden per category op mijn site.

Layouts

In de Layouts map zitten al de bestanden die ik heb gekopiëerd van het thema purehugo. In deze bestanden heb ik het meeste veranderd. Ik heb extra code toegevoegd in de sidebar die zorgt voor een hoofdmenu op de sidebar van mijn site.

Static

In de Static map zit al de css en de javascript van de site. Aan de javascript heb ik zelf niets veranderd maar in de css-file heb ik zelf nog veel code extra bijgeschreven om wat van de opmaak van de site te veranderen.

Config.toml

In dit bestand wordt de site geconfigureerd. Hier staan aanwijzigingen in voor Hugo hoe de site moet worden opgebouwd.

Mijn keuze

Ik verkies hugo boven WordPress. Ookal vond ik het heel moeilijk om te beginnen met hugo om wille van de mappenindeling verkies ik het boven WordPress. WordPress is veel gemakkelijker om te gebruiken maar je hebt niet zo veel mogelijkheden. Alle sites die met WordPress worden gemaakt, lijken op een of andere manier op elkaar.
Ook heb je in hugo meer vrijheid je kan kiezen wat je doet: van een theme beginnen, zelf een theme maken, zelf templates maken…
Tenslotte werkt hugo ook goed samen met GitLab, je commits laten heel goed zien wat je de laatste keer allemaal hebt aangepast.
